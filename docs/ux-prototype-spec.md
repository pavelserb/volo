# Спецификация UX/UI-прототипа Volo

> Документ описывает цели, объём, информационную архитектуру, ключевые сценарии и требования к мок-данным для **интерактивного прототипа** веб-интерфейса. Основа: [concept.md](./concept.md).

---

## 1. Назначение документа

Спецификация нужна, чтобы:

- единообразно понимать **что** входит в прототип и **что** сознательно исключено;
- иметь **чек-лист экранов и состояний** для дизайна или фронтенд-реализации;
- проводить **прогоны сценариев «всухую»** и фиксировать инсайты для бизнес-логики, UX-стратегии и PRD.

Прототип **не** заменяет PRD, юридические тексты и интеграции с платежами/бэкендом.

---

## 2. Цели прототипа

1. **Проверить основную воронку** «посадочная → каталог → карточка SKU → выбор даты/слота → участники/вместимость → опции → контакт/оплата → подтверждение» на ощущаемость и трение.
2. **Отработать сложные UI-паттерны:** shared capacity, остаток мест, опция buyout (выкуп целого слота), exclusive + per slot.
3. **Смоделировать подарочный сценарий:** покупка сертификата, выбор формата выдачи, активация получателем и доведение до бронирования.
4. **Показать неявную регистрацию:** данные собираются в процессе оформления; отдельного шага «зарегистрируйтесь» нет; опционально — точка входа «Войти через Google» / «Войти по ссылке на email».
5. **Задать структуру личного кабинета:** бронирования, подарки (купленные / полученные), отзывы (заглушка), настройки.
6. **Продемонстрировать двухслойную навигацию:** зонтичный бренд Volo + вход с посадочной «город × направление».

---

## 3. Не-цели (вне объёма прототипа)

- Реальная оплата, webhooks, бухгалтерия.
- Реальная доступность слотов от провайдера; синхронизация вместимости между пользователями (гонки за последнее место).
- Полноценный **админ-кабинет** и **кабинет провайдера** — только при необходимости **минимальные** заглушки (см. раздел 11).
- Юридически выверенные тексты (оферта, политика конфиденциальности) — достаточно lorem или кратких плейсхолдеров.
- Локализация «под ключ»: в прототипе достаточно **одного основного языка интерфейса** (рекомендация: **PL** как рынок v1) с возможностью показать **переключатель EN** на ключевых экранах для проверки длины строк.
- PWA, push-уведомления, offline.

## 4. Контекст продукта (зафиксировано из концепции)

| Параметр | Значение для прототипа |
|----------|-------------------------|
| Рабочее имя бренда | Volo |
| Город v1 | Краков |
| Валюта | PLN |
| Языки | PL (основной в прототипе); EN — на выбор для демо переключателя |
| Модель | Branded service; клиент покупает у Volo |
| Ключевые направления для демо | Минимум: **вертолёт** (shared + buyout); опционально: **трансфер** (exclusive + per slot) для контраста |

## 5. Формат и платформа

- **Тип:** responsive **веб** (mobile-first или desktop-first — на выбор исполнителя; обязательно показать **мобильную** ширину для основного flow).
- **Данные:** статические / захардкоженные JSON или константы в коде; даты слотов — фиктивные, но правдоподобные (например, ближайшие 14 дней от «сегодня» прототипа).
- **Навигация:** полноценные переходы между экранами (кликабельно), без реального API.

Рекомендуемая технология на усмотрение команды: HTML/CSS/JS, Figma + прототип-связи, Next.js/React — спецификация **технологически нейтральна**.

## 7. Информационная архитектура (верхний уровень)

```
/                          — главная (зонтичная: доверие, выбор города/направления)
/{city}/{direction}        — посадочная «направление в городе» (SEO-логика)
/{city}/experiences        — каталог услуг города (опционально как отдельный URL)
/s/{sku-slug}              — карточка SKU
/book/{skuId}/…            — воронка бронирования (шаги см. ниже)
/gift/…                    — воронка подарочного сертификата
/redeem                    — ввод кода активации сертификата
/account/…                 — личный кабинет (после «условного входа»)
/login                     — вход (Google / magic link — UI-имитация)
/corporate                 — «Для бизнеса» (лёгкая страница + форма)
/legal/…                   — заглушки правовых страниц
```

ЧПУ и точные пути могут быть упрощены в прототипе; важно сохранить **смысловую** структуру «город × направление → SKU → бронирование».

## 8. Инвентарь экранов и обязательные элементы

### 8.1. Маркетинг и навигация

| Экран / блок | Обязательное содержание |
|--------------|-------------------------|
| Главная | Ценностное предложение Volo; быстрый выбор города (Краков); блоки направлений (вертолёт, трансфер, …); футер с ссылками |
| Посадочная «Краков × вертолётные полёты» | H1 под запрос; краткое УТП; 2–4 карточки SKU; доверие (отзывы агрегированно или «как будет»); CTA в каталог/карточку |
| Шапка сайта | Логотип; город (селектор с зафиксированным Краковом); язык PL/EN; вход в аккаунт; корзина не обязательна |
| Подвал | Контакты-заглушка; ссылки на legal; соцсети опционально |

### 8.2. Каталог и карточка SKU

| Экран | Обязательное содержание |
|-------|-------------------------|
| Каталог города | Фильтры: направление, дата (опционально), бюджет (опционально), количество человек; сетка карточек |
| Карточка SKU | Медиа; название; краткое описание; **цена** (с пояснением per person / per ride); что включено; политика отмен (кратко + ссылка «подробнее»); блок отзывов (2–3 мок-отзыва + рейтинг); CTA «Выбрать дату» / «Подарить» |

### 8.3. Воронка бронирования (основной сценарий)

Единая логика шагов (допускается объединение шагов на одном экране на мобиле):

1. **Выбор даты** — календарь; только дни с доступными слотами (мок).
2. **Выбор времени** — список слотов; для shared — отображение **остатка мест** (например, «Осталось 2 места»).
3. **Участники и вместимость** (для **shared**): stepper или селектор числа человек; валидация «не больше свободных мест».
4. **Buyout** (если SKU в режиме shared с buyout): чекбокс или карточка «Приватный полёт — выкупить все места» с **отдельной ценой** (может отличаться от N × per unit).
5. **Дополнения (add-ons)** — 1–2 опции (например, видеосъёмка) с чекбоксами.
6. **Контактные данные** — имя, email, телефон; без отдельного экрана «Регистрация».
7. **Модель оплаты (визуально)** — для прототипа: «Оплатить 100%» или «Депозит + доплата» в зависимости от выбранного демо-SKU; итоговая сумма.
8. **Оплата** — форма-заглушка (номер карты не собирать реально); кнопка «Оплатить» ведёт к успеху.
9. **Подтверждение** — номер бронирования, дата/время, что делать дальше; кнопки «В аккаунт», «На главную».

**Ветвление подтверждения (имитация):**

- **Instant** — сразу «Подтверждено».
- **Request-based** — экран «Заявка принята, ожидаем подтверждения в течение X часов» (таймер как текст, не реальный).

Отдельно для демо-SKU **exclusive + per slot** (трансфер): шаг «участники» может сводиться к «до N пассажиров» без shared-остатка; слот не дробится между бронированиями.

### 8.4. Подарочный сертификат

| Экран | Обязательное содержание |
|-------|-------------------------|
| Старт подарка | С карточки SKU: «Подарить» → переход в поток подарка |
| Персонализация | Имя получателя; сообщение; опционально «дата вручения» (текстовое поле) |
| Выбор формата выдачи | Минимум три опции: **скачать PDF**, **отправить на email получателя**, **скопировать ссылку** (показ мок-ссылки + toast «скопировано») |
| Оплата | Аналогично бронированию (заглушка) |
| Успех (покупатель) | Подтверждение заказа; кнопки повторить действие с сертификатом (скачать / ссылка) |

**Получатель:**

| Экран | Обязательное содержание |
|-------|-------------------------|
| /redeem | Поле кода; после ввода — экран «Ваш подарок: [название SKU]» |
| Бронирование из сертификата | Тот же flow выбора даты/времени/участников, что и обычное бронирование; в шапке пометка «По подарочному сертификату» |

### 8.5. Аутентификация (UI)

| Экран | Обязательное содержание |
|-------|-------------------------|
| /login | Кнопка «Продолжить с Google» (мок); поле email + «Выслать ссылку» (мок); пояснение «без пароля» |
| После «входа» | Редирект в аккаунт или на предыдущий экран (имитация query `?next=`)

### 8.6. Личный кабинет

| Раздел | Обязательное содержание |
|--------|-------------------------|
| Мои бронирования | Список 2–3 мок-бронирований: предстоящие / прошлые; статусы (подтверждено, ожидает подтверждения) |
| Подарки | Подвкладки или фильтры: **Я подарил** / **Мне подарили**; статусы: ожидает активации / активирован / использован / истёк (мок-данные) |
| Мои отзывы | Заглушка: «Отзывы появятся после поездок» или один мок-отзыв |
| Настройки | Имя, email, телефон; язык; кнопка «Выйти» (мок) |

### 8.7. Корпоративная страница (v1 лёгкий формат)

- Статический контент + форма (имя компании, контакт, комментарий) — отправка **не** реальная; сообщение «Спасибо, мы свяжемся».

### 8.8. Юридические заглушки

- `/legal/terms`, `/legal/privacy` — короткий текст-заглушка.

## 9. Мок-данные (минимальный набор)

Рекомендуется завести один файл или константу «демо-каталог»:

### 9.1. SKU A — вертолёт, shared + buyout + per unit

- Название (PL): например, «Обзорный полёт над Краковом, 20 минут».
- Направление: вертолёт; город: Краков.
- **Вместимость слота:** 3 места; **shared**; **buyout** включён.
- Цена: **per unit** (например, 899 PLN / чел.); **цена buyout** (например, 2399 PLN за весь слот — не обязательно 3×899).
- Слоты: не менее **5** разных временных окон на разные дни; у части слотов разный `остаток мест` (3, 1, 2).

### 9.2. SKU B — трансфер, exclusive + per slot

- Название: «VIP-трансфер: аэропорт ↔ центр».
- **Exclusive**; цена **за поездку**; до 4 пассажиров без влияния на цену.
- Слоты: реже, чем у вертолёта (имитация ограниченной доступности).

### 9.3. Отзывы

- 3 отзыва на SKU A с разными рейтингами; 1–2 на SKU B.

### 9.4. Пользователь / бронирования / подарки

- Текущий «вошедший» пользователь — мок: email `demo@volo.local`.
- В аккаунте: 1 предстоящее бронирование, 1 прошлое; 1 подарок «я подарил» (ожидает активации); 1 «мне подарили» (мок).

## 10. Состояния и ошибки (для «всухую»)

Имеет смысл заложить **по одному экрану/модалке** (даже статичной):

| Ситуация | Поведение прототипа |
|----------|---------------------|
| Нет слотов на выбранную дату | Пустое состояние + CTA «Другая дата» |
| Выбрано больше мест, чем осталось | Блокировка кнопки «Далее» + сообщение |
| Платёж не прошёл (имитация) | Редкий сценарий: кнопка «Симулировать ошибку» в dev или отдельная ветка |
| Код сертификата неверный | Сообщение об ошибке на /redeem |

---

## 11. Кабинет провайдера и админка (опционально, фаза B)

**По умолчанию не входит в MVP прототипа.** Если время есть:

- **Провайдер:** список заказов (2–3 строки) + кнопки «Подтвердить» / «Отклонить» (мок); экран «Доступность» — упрощённая таблица слотов без реальной логики.
- **Админ:** одна страница «Очередь бронирований» — заглушка.

---

## 12. Мультиязычность в прототипе

- Все ключевые экраны воронки бронирования и подарка — **на польском**.
- Переключатель **EN** на шапке: на 1–2 экранах показать **удлинение строк** (проверка вёрстки); полный перевод всего сайта не обязателен.

---

## 13. Критерии готовности прототипа (Definition of Done)

Прототип считается достаточным, если:

1. Можно пройти **полный путь** по SKU A от посадочной или главной до экрана подтверждения оплаты.
2. Реализованы **shared + выбор числа людей + buyout** (хотя бы на одном SKU).
3. Реализован **подарок**: персонализация → выбор формата (PDF / email / ссылка) → успех → **redeem** → бронирование.
4. Показан **личный кабинет** с разделами бронирования и подарков (мок-данные).
5. На карточке SKU видны **отзывы и рейтинг** (мок).
6. Интерфейс **читабелен на мобильной ширине** (320–390px) для основного flow.

---

## 14. Что фиксировать после прогонов

В ходе тестов рекомендуется вести список:

- непонятных формулировок и «лишних» шагов;
- вопросов к бизнес-логике (таймауты, гонки за места, депозит vs 100%);
- гипотез для A/B (порядок шагов, buyout как отдельный шаг vs на одном экране со слотом);
- решений по **единицам цены** в UI (как показывать «от», «за человека», «за полёт»).

Эти записи затем переносятся в документ бизнес-логики, UX-стратегию и PRD.
